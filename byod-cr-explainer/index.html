Why BYOD projects couldn't boot — and the 3 CRs that fix it

Stencil, Cloudscape, and Books design-system projects showed "This project failed to load." Here's the root cause and the fix.

TL;DROne root cause, three code paths

Every failure came down to the same mistake in three different functions: each tried to locate (or validate) a design system's template archive without asking the registry — either by hard-coding node22 in the archive name (wrong for the node20 design systems), or by reading the version-root manifest.json straight from S3 (a raw-S3 GET returns 403, because the cross-account grant only covers the …/template/ prefix, not the version root). The fix is the same in all three: ask the Design Registry API for the exact archive location, and stop guessing.

These CRs don't touch Protozoa. They fix the install / boot path. But a project must boot & build before it can be shared — so fixing boot is what unblocks uploading Cloudscape / Stencil / BDS prototypes to Protozoa (this page is one such prototype).

◆The new design — ask the registry, don't guess

API-resolve (the new part) existing flow stage
The Projects-API Lambda asks the Registry API for the exact archive location (③), gets back an s3Uri, and the container SigV4-fetches that specific zip from S3 (④→⑤). The registry decides the node major — no hard-coded list, nothing to probe.

◆"Why probe at all? Can't we just read the node version?"

Exactly the right question — and the answer is yes, we now just ask. Here's the evolution:

ApproachHow it learned the node majorProblem
v0 — original Didn't. Hard-coded node22. Every node20 DS (Stencil/CS/BDS) → 403 → "failed to load."
v1 — probe Tried reading manifest.json from raw S3; that 403'd (grant is …/template/* only, not the version root), so it guessed by HEAD-probing [node22, node20]. Works, but a hard-coded candidate list — a future node24 would silently break again.
v2 — API-resolve ✅ Asks the registry API (api.registry.design.amazon.dev), which returns the exact template.archive.s3Uri. The Lambda role already has execute-api on that API. Authoritative. No guessing, no hard-coded list, future node majors just work.
Why not just read manifest.json from S3? Two ways exist to read that manifest: the raw S3 object at the version root (currently 403s — the cross-account grant only covers …/template/*; we could widen it since we own DesignRegistryCDK), or the registry API (already reachable, returns the same data). We chose the API even though widening was on the table: the API is the registry's published contract, while the raw S3 key layout is an internal storage detail that could change under us. Asking the API means we depend on the interface, not the bucket's shape — and no new grant is needed.

◆The three CRs

CR-289839650
DesignAssistantShared · templateResolver.ts

Resolve the template archive from the registry API

resolveTemplateArtifactUri hard-coded archive.al2023-x64-node22.zip. Stencil/Cloudscape/BDS ship node20, so their URL pointed at a nonexistent object → the container's SigV4 fetch 403'd → "failed to load." Now it calls the registry API (GET /api/packages/{pkg}/{ver}, SigV4 execute-api), reads the exact template.archive.s3Uri, and returns that S3 URL — no node-major guessing, future-proof for new node majors. Distinct error types: TemplateArchiveNotFoundError (unregistered / no zip) vs. RegistryApiError (transient infra).

async — both call sites awaited 15 unit tests no hard-coded node list
CR-289839745
DesignAssistantCDK · projectsApiLambdaManager.ts

Grant the Projects-API Lambda registry-API access

The resolver now calls the registry API, so the Projects-API Lambda role needs execute-api:Invoke on the DRB API gateway (arn:aws:execute-api:us-east-1:137769144454:*/prod/GET/*) — the same grant the mde-environment (container) role already carries. No S3/KMS grant needed on the Lambda anymore; the container keeps its own S3 grant for the actual download.

IAM only execute-api (not S3)
